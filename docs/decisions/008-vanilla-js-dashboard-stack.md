# ADR 008 — Vanilla JS Dashboard Stack

**Date:** 2025-10-15 (decision made at dashboard inception; recorded retrospectively 2026-06-12)
**Status:** Accepted (retrospective)
**Deciders:** Project principal

---

## Context

The project needs a metrics dashboard to visualise pipeline health, filter telemetry, and dataset
growth over time. The dashboard is a read-only display of pre-computed JSON metrics
(`all_metrics.json` generated by `scripts/ops/export_dashboard_data.py`). It is deployed as a
static site to GitHub Pages from `src/dashboard/`.

---

## Decision

Use **vanilla JavaScript ES6 modules** with **Chart.js** for visualisation and **Vite** as the
bundler/dev server. No JavaScript framework (React, Vue, Svelte).

---

## Rationale

1. **Dependency surface:** A dashboard for a research data pipeline is a low-complexity UI.
   A full framework adds hundreds of transitive dependencies, a virtual DOM, and a component
   lifecycle for a problem that does not need them.
2. **Static hosting:** GitHub Pages serves static files; no server-side rendering is needed.
   Vanilla JS + Vite produces a minimal, framework-free bundle that loads quickly on Pages.
3. **Maintainability:** Any contributor who knows JavaScript can modify the dashboard without
   learning a framework's conventions. This aligns with the project's low-resource NLP focus —
   the primary contributors are ML/data engineers, not front-end specialists.
4. **Playwright compatibility:** Playwright's browser automation tests work identically for vanilla
   JS and framework-based dashboards; there is no testing tooling benefit to adding a framework.
5. **Live reload dev server:** `somali-tools dashboard serve` provides a live-reload dev server
   for local iteration without a full build step.

---

## Consequences

- Dashboard source lives under `src/dashboard/src/` (TypeScript/JS modules).
- Built artifacts go to `src/dashboard/_site/` (gitignored); the deploy workflow publishes `_site/`.
- `playwright.config.js` is at the repository root (Playwright's default discovery location),
  not under a dedicated `config/` directory.
- `docs/reference/metrics_schema.json` is the JSON-Schema contract for `all_metrics.json`;
  `dashboard-validation.yml` validates the dashboard data against it on every push.
- If the dashboard requires significant interactive complexity in the future (e.g., annotation
  tooling for Stage 2 labelling), this decision should be revisited.

---

## Alternatives Considered

| Alternative | Reason rejected |
|-------------|-----------------|
| React | High complexity overhead for a read-only metrics display; large bundle size |
| Vue | Same reasoning as React |
| Svelte | Smaller bundle but still a compilation step with different learning curve |
| Observable Plot / Vega-Lite | Better fit for interactive exploration notebooks; less suitable for a production dashboard with a custom layout |
| Grafana (embedded) | Requires a running Grafana instance; conflicts with the static GitHub Pages deployment model |
